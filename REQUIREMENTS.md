# Web Cursor · AI React Playground 需求文档

> 版本 v0.2 ｜ 本文件只定义"做什么 / 为什么 / 分期",不涉及技术实现。
> 技术基线(已定):Next.js(App Router)+ Monaco + esbuild-wasm(浏览器内转译)+ iframe 沙箱;LLM=DeepSeek 经服务端代理。
> 二期起新增持久化基线:Postgres(Neon)+ drizzle-orm + zod;详见 `docs/backend-design.md`。
>
> 变更记录:v0.2(2026-06-20)二期决策拍板——"无后端持久化"非目标移出(改为 Postgres/Neon 持久化);多文件升级为"真·多文件执行";新增"历史与项目管理"。

## 一、产品定位与目标

**一句话**:一个浏览器内的 AI React 编码沙箱——用自然语言描述需求,AI 写代码,即时执行渲染,**并能读取运行结果自主修复**。

**要解决的核心问题(按优先级)**
1. 让不写完整代码的人,用一句话得到一个能跑、能看的 React 界面。
2. 让 AI 不只是"生成代码",而是"对生成结果负责"——跑挂了自己改。
3. 把一次性生成,变成可对话、可迭代的持续编辑。

**衡量做成了没有(产品级指标)**
- 北极星:**一次需求 → 可运行结果的成功率**(用户描述需求后,最终拿到能正常渲染界面的比例)。
- 辅助:平均自我修复轮次、从输入到首次渲染的时长、多轮迭代留存。

---

## 二、目标用户与场景

- **主用户**:想快速验证 UI 想法的前端 / 设计 / 产品,不想从零搭环境。
- **典型场景**
  1. 描述一个组件 / 页面,立刻看到效果。
  2. 看到效果后用自然语言微调("按钮改蓝""加个删除")。
  3. AI 生成出错时,不用我懂报错,它自己修。
- **非用户**:需要完整工程 / 部署生产代码的团队协作开发(明确不服务)。

---

## 三、非目标(防止做飞)

- ❌ 真鉴权 / 账号体系 / 多人协作(二期用匿名 owner id 做数据隔离,**非鉴权**;`x-owner-id` 谁都能伪造,公网多租户上线前必须补真鉴权)
- ❌ 支持任意 npm 包(固定依赖;WebContainers 仅作远期备选)
- ❌ 多语言 / 多框架(只做 React,做深不做广)
- ❌ 移动端适配(桌面优先)

> 📌 **已移出非目标(二期纳入):后端数据库持久化。** 原"代码存浏览器本地即可"在引入"多轮对话记忆 + 历史会话"后不再成立——对话上下文与 transcript 必须由后端权威存储(前端不可信、不能持有/拼上下文)。落地用 Postgres(Neon),详见 `docs/backend-design.md`。

## 四、一期:核心闭环(MVP)

> **目标**:证明灵魂功能——AI 写代码、能跑、跑挂了自己修,并能把成果带走。
> 做完这一期,产品从"玩具"变成"AI coding agent"。

**用户故事**
- 作为用户,我能在输入框描述需求,得到一个 AI 生成、立即渲染的 React 界面。
- 作为用户,当 AI 代码出错时,我无需理解报错,系统能自动让 AI 修到跑通。
- 作为用户,代码出错时我能看到清晰的错误提示,而不是白屏。
- 作为用户,我做出满意的界面后,能导出一个可独立打开的 HTML 文件。

**需求清单**

| 编号 | 需求 | 说明 |
|---|---|---|
| R1 | 自然语言生成代码 | 输入需求 → AI 生成 React 代码 → 自动渲染 |
| R2 | 即时执行与预览 | 代码变化后实时渲染结果(沿用现有沙箱) |
| R3 | 执行结果可观测 | 系统能感知:渲染成功 / 运行报错 / 编译报错 / 卡死 |
| R4 | 错误可见 | 用户能看到错误浮层,而非白屏(脱离 AI 也独立有用;是 R5 的前置) |
| R5 | **AI 自我修复** ⭐ | 把执行结果(报错)反馈给 AI,自动迭代修复,循环至跑通或达上限 |
| R6 | 修复过程可见 | 用户能看到"AI 正在第 N 次尝试修复"的状态,不是黑盒卡住 |
| R7 | 导出为静态 HTML | 把当前结果导出成自包含 .html 文件,脱离本应用、在浏览器打开即可运行 |

**一期范围限定**:单文件、固定依赖、生成完成后整体渲染(非流式)。

**一期 Demo 验收**:输入"做一个待办列表",AI 生成代码,人为引入一个 bug,**亲眼看到 AI 读报错、自己改、最终跑通**;并能导出一个可独立打开的 HTML。

## 五、二期:产品化体验

> **目标**:从"能跑的 demo"变成"用着像产品"——能记住、能多文件、能对话、流式顺滑。

**用户故事**
- 作为用户,我能连续对话迭代("再加个筛选""换个配色"),AI 记得上下文、基于现有代码改。
- 作为用户,我刷新 / 关掉再打开,之前的项目、对话、代码都还在,能接着聊。
- 作为用户,我能在多个项目之间切换;同一项目内可开多条对话线索(都对着该项目的代码),各自独立。
- 作为用户,我能让 AI 构建包含多个组件 / 文件的应用,并看到文件树。
- 作为用户,我能看着代码一边生成一边出现,而不是干等。

**需求清单**

| 编号 | 需求 | 说明 |
|---|---|---|
| R8 | 多文件项目(真执行) | 文件树 + 多 tab 编辑;文件间可 `import` 互相引用;转译从单文件 `transform` 升级为 **bundle(虚拟 FS)**,本地模块不再走 importmap |
| R9 | 对话式迭代(有状态) | 多轮上下文由**后端**按会话 transcript 重建;基于现有代码增量改,而非每次重生成;**前端不持有、不拼上下文** |
| R10 | 流式生成 | AI 边生成边显示代码 / 渐进应用 |
| R11 | 迭代不丢状态 | 微调代码后,运行中的组件状态尽量保留(不归零) |
| R12 | 代码可手改 | 用户能手动编辑 AI 生成的代码,改完即时生效并存盘 |
| R13 | 安全隔离 | 不可信 AI 代码与主站隔离(独立 origin / 沙箱属性) |
| R19 | 持久化基座 | 项目 / 文件 / 会话 / 消息存 Postgres(Neon);匿名 owner id 隔离;`role=tool` 报错也入库(修复过程可回放)。**Cursor 模型:项目=共享代码库,项目下多条会话线索**(代码挂项目、会话间共享;messages 挂会话)。详见 `docs/backend-design.md` |
| R20 | 历史与项目管理 | 首页项目列表;项目内会话列表 + 切换 / 新建;打开会话按 `seq` 回放 transcript、接着聊;**切会话只换聊天记录、代码/预览不变**;"新会话" = 同项目 chat 懒建。项目改名 / 删除 |

> 注:R19/R20 是二期新增需求,编号续在三期 R18 之后,**不属于三期**;归属见下方拆分。

**二期拆分(已拍板,分批落地,各自独立 OpenSpec change)**
- **change 1 — 持久化 + 多轮 + 历史**:R19(持久化基座) + R9(有状态多轮) + R20(历史与项目管理 UI)。依赖已设计的 `docs/backend-design.md`,改造集中在后端 + B 域编排,风险低。**先做。**
- **change 2 — 真·多文件执行**:R8(多文件) + 转译 bundle 改造 + 文件树 / 多 tab UI。风险集中在转译 / 沙箱层,独立验收。**后做。**
- R10–R13(流式 / 状态保留 / 手改 / 隔离)随两个 change 自然覆盖,不单列里程碑。

**二期增强**:R7 导出升级为"完全离线自包含"(依赖内联,断网 `file://` 可跑);R7 支持多文件项目导出。

**二期 Demo 验收**
- change 1:开两个项目各聊几轮,人为引 bug 看 AI 自修;**刷新页面 → 项目 / 会话 / 代码全在,接着聊上下文不丢**。
- change 2:连续对话搭一个多文件小应用(组件互相 import),流式看到生成、手改一行即时生效。

## 六、三期:差异化亮点(远期)

> **目标**:做出别人 demo 里没有的深度,撑起技术稀缺性。

**需求清单**

| 编号 | 需求 | 说明 |
|---|---|---|
| R14 | 真实运行时(WebContainers) | 执行层升级,支持真 `npm install` / 真 dev server,可用任意包(**备选,工作量大 + 部署天坑**) |
| R15 | 版本与回滚 | 每次 AI 修改可作为 checkpoint,能回退任意历史版本 |
| R16 | 读码再改 | AI 能理解已有代码库再修改,而非只会从零生成 |
| R17 | 分享与部署 | 一键分享链接 / 部署(R7 静态导出的进阶分发能力) |
| R18 | 任务规划 | 复杂需求 AI 先拆解计划再执行,过程可见 |
